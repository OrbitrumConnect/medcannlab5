## 🔧 CORREÇÕES APLICADAS — ANÁLISE TÉCNICA E EDUCATIVA

### 1. WiseCare Auth: De 401 a Bearer Token (commit `71b004c`)

**Contexto histórico:** No dia 03/03, integramos a WiseCare API. O Edge Function `wisecare-session` foi deployado com autenticação Basic Auth, que era o padrão documentado. Porém, o servidor homologação retornava:

```json
{"title": "Missing authorization header", "code": "MISSING_AUTHORIZATION_HEADER"}
```

**Root cause:** A API de autenticação WiseCare usa um endpoint **diferente** da API de dados:
- **Auth:** `POST https://session-manager.homolog.v4h.cloud/api/auth/org/` (sem `/v1/`)
- **API:** `GET/POST https://session-manager.homolog.v4h.cloud/api/v1/...` (com `/v1/`)

**Lição técnica:** APIs REST nem sempre seguem versionamento uniforme. O endpoint de autenticação frequentemente vive em um namespace separado do namespace de dados. Isso é padrão em APIs enterprise (OAuth2, por exemplo, usa `/oauth/token` enquanto dados ficam em `/api/v2/`).

**Lição arquitetural:** A abordagem de fallback multicamada (Bearer → Basic → UUID como token) que implementamos no dia 03/03 era correta em princípio, mas o endpoint estava errado. A confirmação de Leonardo foi essencial — **documentação nem sempre é suficiente; validação com o fornecedor é obrigatória**.

### 2. Admin Universal: De Emails Hardcoded a RBAC Real (commits `b881e53`, `0b51561`)

**O problema filosófico:** A RPC `admin_get_users_status` e as RLS policies verificavam acesso administrativo assim:

```sql
-- ANTES (frágil, insustentável)
AND (auth.jwt() ->> 'email') IN ('phpg69@gmail.com', 'admin@medcannlab.com')
```

Isso é o que chamamos de **"security by enumeration"** — o sistema conhece cada admin pelo nome. Funcionava quando Pedro era o único admin. Quebrou no momento em que Ricardo, João e Eduardo precisaram acessar.

**A solução correta (RBAC):** Substituímos por:

```sql
-- DEPOIS (escalável, correto)
IF NOT public.is_admin() THEN
    RAISE EXCEPTION 'Acesso Negado';
END IF;
```

A função `is_admin()` (criada em 19/02, migration `20260219011138`) consulta a tabela `user_roles`. Para adicionar um novo admin, basta um INSERT na tabela — zero mudança de código.

**Conceito educativo — RBAC (Role-Based Access Control):**
Em sistemas de saúde, o modelo RBAC é obrigatório. A ANVISA e a LGPD exigem que o acesso a dados clínicos seja determinado pela **função do profissional**, não pela identidade pessoal. O RBAC implementado no MedCannLab segue 4 princípios:

1. **Princípio do Menor Privilégio:** Cada usuário tem exatamente as permissões necessárias para sua função.
2. **Segregação de Deveres:** Quem prescreve (profissional) não é quem audita (admin).
3. **Revogabilidade:** Remover acesso é tão simples quanto deletar uma linha em `user_roles`.
4. **Auditabilidade:** A tabela `user_roles` tem `granted_at` — sabemos quando e para quem cada role foi concedida.

### 3. Type Mismatch: varchar(255) vs text (commit `0b51561`)

**O problema:** PostgreSQL é estrito com tipos de retorno em funções. A tabela `users.name` é `varchar(255)`, mas a função declarava `RETURNS TABLE(name text)`. O PostgreSQL rejeitou com erro `42804`.

**Lição técnica:** Em PostgreSQL, `varchar(n)` e `text` são tipos distintos internamente, mesmo que ambos armazenem strings. Quando uma função `RETURNS TABLE(...)` declara um tipo, o PostgreSQL exige correspondência exata. O cast `u.name::text` resolve sem custo de performance.

**Observe:** Este erro já existia antes da nossa modificação. A versão com emails hardcoded tinha o mesmo bug — mas como o email do Pedro estava na whitelist, ele nunca via o erro. Ricardo via porque não estava na lista, então o erro do email mascarava o erro do type. Corrigir o acesso revelou o bug subjacente.

### 4. Carolina Campello: O Gap de Sincronização

**Cenário:** Carolina (filha do Dr. Ricardo) fez signup no app em 25/02/2026. O Supabase Auth criou seu registro em `auth.users` com `meta_type: patient`. Porém, **nenhum registro foi criado em `public.users`**.

**Consequência:** Para o sistema, Carolina não existia. Não aparecia no painel de pacientes, nem no admin. Era um "paciente fantasma" — autenticada mas invisível.

**Root cause:** Não existia trigger `AFTER INSERT ON auth.users` para sincronizar novos signups com `public.users`.

**Reflexão arquitetural:** Este é um padrão clássico chamado **"dual-write problem"**. Quando dois sistemas (auth e application) precisam estar sincronizados, a falha de um write gera inconsistência. A solução correta é usar um **trigger de banco** (que garante atomicidade) em vez de depender do frontend para fazer dois INSERTs separados.

**O trigger criado:**
```sql
CREATE TRIGGER on_auth_user_created
  AFTER INSERT ON auth.users
  FOR EACH ROW
  EXECUTE FUNCTION public.handle_new_user();
```

**Garantias:**
- `SECURITY DEFINER` — roda com permissões do owner
- `SET search_path = public` — anti-injection
- `ON CONFLICT DO UPDATE` — idempotente (se já existir, atualiza)
- Cria automaticamente a `user_role` baseada no `meta_type` do signup

**Lição do Livro Magno:** *"A integridade de dados não pode depender do frontend. Se o banco pode garantir a consistência sozinho, ele deve."*

## 🔄 FLUXOS DOCUMENTADOS

### Fluxo: Profissional Adiciona Paciente

```
1. Profissional acessa NewPatientForm
2. Preenche: nome, email, CPF, data nascimento
3. handleSubmit() → INSERT em public.users (type='patient')
4. INSERT em clinical_assessments (doctor_id = profissional.id)
5. getAllPatients() → busca via clinical_assessments.doctor_id
6. Paciente aparece no painel do profissional ✅
7. Admin vê via admin_get_users_status() (JOIN public.users + auth.users) ✅
```

**Limitação conhecida:** O paciente cadastrado pelo NewPatientForm NÃO recebe email nem notificação. Ele só existe na `public.users` (sem `auth.users`), então não consegue fazer login sozinho. Ele precisaria fazer **self-signup** separadamente.

### Fluxo: Paciente Self-Signup (CORRIGIDO HOJE)

```
1. Paciente acessa app → tela de Login → "Criar Conta"
2. Supabase Auth → INSERT em auth.users
3. TRIGGER on_auth_user_created → INSERT em public.users (sync)
4. TRIGGER → INSERT em user_roles (role baseada no meta_type)
5. Paciente visível em TODO o sistema ✅
```

**Antes do trigger:** Passo 3 e 4 não existiam. O caso da Carolina mostrou essa falha.

### Fluxo: Video Call (Dual-Provider)

```
1. Usuário aceita chamada
2. VideoCall.tsx → POST /wisecare-session (Edge Function)
3. Edge Function → POST /api/auth/org/ (Bearer token)
4. Edge Function → POST /api/v1/rooms (criar sala)
5. Se WiseCare OK → iframe com joinUrl (badge verde 🟢)
6. Se WiseCare falha → WebRTC P2P automático (badge amarelo 🟡)
```
