## 🔧 CORREÇÕES APLICADAS — ANÁLISE TÉCNICA

### 1. FK `chat_participants_user_id_fkey`: Cross-Schema Violation

**Problema:** A tabela `chat_participants.user_id` referenciava `auth.users(id)`, mas pacientes cadastrados via `NewPatientForm` existiam APENAS em `public.users`. Resultado: FK violation ao criar salas de chat.

```
ERROR: 23503: insert or update on table "chat_participants" 
violates foreign key constraint "chat_participants_user_id_fkey"
DETAIL: Key (user_id)=(5a83ab19-...) is not present in table "users".
```

**Fix (2 etapas):**
```sql
-- Etapa 1: Sincronizar orphans
INSERT INTO public.users (id, email, name, type)
SELECT au.id, au.email, COALESCE(au.raw_user_meta_data->>'name', ...), ...
FROM auth.users au WHERE au.id IN (
  SELECT cp.user_id FROM chat_participants cp 
  WHERE cp.user_id NOT IN (SELECT u.id FROM public.users u)
);

-- Etapa 2: Trocar FK
ALTER TABLE chat_participants DROP CONSTRAINT chat_participants_user_id_fkey;
ALTER TABLE chat_participants ADD CONSTRAINT chat_participants_user_id_fkey 
  FOREIGN KEY (user_id) REFERENCES public.users(id) ON DELETE CASCADE;
```

**Lição:** FKs cross-schema (`public` → `auth`) são perigosas porque os dados podem existir em um schema mas não no outro. O modelo correto é: `public.users` é a tabela master; `auth.users` é gerenciada pelo Supabase Auth.

### 2. Admin Invisível: O Paradoxo auth.uid()

**Problema:** Os 4 admins (Pedro, Ricardo/Nôa, Eduardo, João) estavam autenticados via `auth.users` mas **NÃO tinham registro** em `public.users`. O trigger `on_auth_user_created` foi criado em 04/03 mas DEPOIS do signup deles.

**Consequência:**
```sql
is_admin()             → false  (checa user_roles → vazio)
is_admin_user()        → false  (checa public.users.type → inexistente)
get_current_user_type() → 'patient'  (fallback padrão)
```

**Curiosidade:** O frontend funcionava porque lia de `auth.users.raw_user_meta_data`, mas o RLS (que usa `auth.uid()` + funções SQL) não encontrava os admins.

**Fix:** INSERT com CTE para os 4 admins em `public.users` + `user_roles`.

**Descoberta adicional:** `auth.uid()` retorna `null` no SQL Editor do Supabase — isso é normal pois roda como service_role (postgres), não como o usuário autenticado. Funções que dependem de `auth.uid()` só podem ser testadas via frontend.

### 3. Dropdown de Pacientes: De Query Direta a RPC Inteligente

**Problema:** `RicardoScheduling`, `EduardoScheduling` e `ProfessionalScheduling` faziam:
```typescript
// ANTES — sujeito a RLS, só via Carolina
supabase.from('users').select(...).eq('type', 'patient')
```

**Fix — Usar `getAllPatients()` de `adminPermissions.ts`:**
```typescript
// DEPOIS — admin vê todos, profissional vê vinculados
const patients = await getAllPatients(user)
```

**Comportamento:**
- **Admin** → retorna todos os 22 pacientes
- **Profissional** → retorna apenas pacientes com vínculo (via `appointments.professional_id` ou `clinical_assessments.doctor_id`)

**Arquivos modificados:**
- `src/components/RicardoScheduling.tsx` — import + `loadPatientsList()`
- `src/components/EduardoScheduling.tsx` — import + `loadPatientsList()`
- `src/pages/ProfessionalScheduling.tsx` — import + `loadData()`

## 🔍 AUDITORIA PROFUNDA — ÁREAS QUE FALTAM

### 🔴 CRÍTICO (Bloqueia lançamento)

#### 1. Pagamentos (30% — ~4-6h de trabalho)
- **Estado:** Nenhum gateway implementado
- **Necessário:** Stripe ou Mercado Pago
- **Tabelas:** `payments`, `subscriptions`, `invoices` (não existem)
- **Páginas afetadas:** `PatientFinancialDashboard.tsx` (usa dados mock)
- **Estimativa:** Criar tabelas + RLS + integração frontend + Edge Function

#### 2. Notificações Reais (20% — ~3-4h)
- **Estado:** `notificationService.ts` existe mas é local (browser only)
- **Necessário:** Email quando profissional agenda paciente, WhatsApp API, Push notifications
- **Edge Function `send-email`** existe mas não é chamada de triggers

#### 3. RLS Users Table — Limpeza de Policies (⚠️ Dívida técnica)
- **6 policies SELECT** redundantes na tabela `users`
- **3 funções admin** diferentes: `is_admin()`, `is_admin_user()`, `has_role()`
- **Mistura de idiomas:** policies usam `'paciente'`/`'profissional'` mas DB armazena `'patient'`/`'professional'`
- **INSERT sem WITH CHECK:** 2 policies INSERT com `qual: null` (abertas)
- **UPDATE sem WITH CHECK:** `users_update` não tem WITH CHECK
- **Estimativa:** ~1-2h para consolidar

### 🟡 IMPORTANTE (Deve ser feito antes do lançamento)

#### 4. Validação de Duplicatas no `NewPatientForm` (~1h)
- O formulário de cadastro de paciente não verifica se email já existe

#### 5. Certificados Automáticos (~2h)

#### 6. Gamificação — Frontend → Backend (~2h)

- Precisa: Edge Function ou RPC para `award_points` / `grant_achievement`

#### 7. WiseCare — Teste Real com Salas (~1h)
