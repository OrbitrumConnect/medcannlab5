## 📡 PARTE 2: ANÁLISE RAG — GPT vs REALIDADE

### O Que o GPT Disse Sobre RAG

> *"O sistema não escolhe significado, ele escolhe similaridade matemática."*

### O Que o Core Realmente Faz (L1960-1988)

```typescript
// RAG REAL no tradevision-core:
const keywords = message.split(' ')
    .filter(w => w.length > 3)
    .slice(0, 5);
const queryFilters = keywords.map(w => `conteudo.ilike.%${w}%`).join(',');
const { data: knowledge } = await supabaseClient
    .from('base_conhecimento')
    .select('titulo, conteudo')
    .or(queryFilters)
    .limit(3);
```

| Aspecto | GPT Descreveu | Realidade no Core |
|:--------|:-------------|:------------------|
| Chunks/vetores | Sim (genérico) | ❌ **Não usa vetores** — usa `ilike` (text match) |
| Embeddings | Implícito | ❌ **Não existe** — busca por substring |
| Tabela fonte | Genérica | `base_conhecimento` (título + conteúdo) |
| Limite | Genérico | 3 artigos, 5 keywords máximo |
| Sofisticação | Alta (sugeriu) | Média — funciona mas não é semântica |

### Veredito RAG

> O GPT descreveu RAG *teórico* (embeddings, vetores). O core usa RAG *prático* (SQL ilike). Funciona para o caso de uso atual, mas é um ponto de evolução futura.

## 🧠 PARTE 11: CLINICAL SCENARIO ENGINE — A Filosofia Anti-Fragmentação

### O Debate (Ricardo × GPT)

Ricardo levantou uma questão arquitetural profunda:

> **"Se o cenário for viciado num prêmio ou numa cosmologia de doença, você não evolui, não está apresentando nada de novo no mercado. O que tem de novo é uma IA que colhe história e não categoriza por doença."**

### O Modelo Clássico (que Ricardo REJEITA)

```
entrada → suspeita de doença → classificação → tratamento
```

Isso **fragmenta** o paciente em silos de especialidade (nefrologia, cardiologia, neurologia).

### O Modelo MedCannLab (que Ricardo PROPÕE)

```
história clínica bruta
    ↓
extração semântica
    ↓
vetor de risco (não vetor de doença)
    ↓
gerador de cenários
    ↓
mapeamento opcional para doenças (label, não estrutura)
```

> **A doença é hipótese, não ponto de partida.**

### Verificação vs Código Atual

| Princípio Ricardo | Implementado? | Onde |
|:-----------------|:-------------:|:-----|
| IA colhe história sem categorizar | ✅ | AEC 001 etapas 1-5: narrativa pura, sem rótulos |
| Doença não é o centro | ✅ | `CLINICAL_PROMPT` L1691-1692 |
| Cenário antes do diagnóstico | ⚠️ Parcial | COS gera vetores, mas ainda não gera cenários múltiplos |
| Organizar por sistemas fisiológicos, não especialidades | 🔮 Futuro | Não implementado — camada semântica ainda não existe |

### Conexão com TradeVision Core

GPT identificou corretamente que o motor nasceu para:

| Trading (origem) | Clínica (adaptação) |
|:-----------------|:-------------------|
| Preço, volatilidade, volume, macro | Dados clínicos, biomarcadores, história |
| Cenários de mercado | Cenários clínicos |
| Risco financeiro | Risco clínico |

### Organização Futura Recomendada

```
❌ Errado (por especialidade):     ✅ Correto (por sistema fisiológico):
semantic/                          semantic/
  nephrology/                        inflammation/
  cardiology/                        metabolism/
  neurology/                         neuro/
                                     vascular/
                                     immune/
```

> **Conceito-chave**: O sistema está a caminho de um **Clinical Scenario Engine** — raro no mercado. A maioria faz apenas: questionário → guideline → resposta.

### Impacto no Plano de Fixes

Esta filosofia **não muda os fixes #1-8** (que são pragmáticos). Mas define a DIREÇÃO futura:
- Fix #9 (Script DRC) deve seguir modelo de **cenários**, não de **doença**
- Camada semântica futura organizada por **sistemas fisiológicos**
- Core continua como **roteador**, não como árvore clínica
