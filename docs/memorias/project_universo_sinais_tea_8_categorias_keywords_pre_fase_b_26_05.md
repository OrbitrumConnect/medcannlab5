## 5 cuidados empíricos críticos

### 1. NÃO inferir diagnóstico TEA da extração

Aplicação direta de [[feedback_locks_macro_vs_micro_matrix_alucinacao_completiva_25_05]]: sidecar mostra **SINAIS LITERAIS detectados**, não diz *"paciente provável TEA"*. Eduardo (neurologista) é quem interpreta.

Igual sidecar renal mostra *"CKD-EPI 30.7"* mas não diz *"paciente é doente renal"* — Ricardo é quem decide.

### 2. Falso positivo "agressividade"

Adulto pode falar *"tenho explosões de raiva"* sem ser TEA — pode ser TDAH, transtorno bipolar, abstinência, depressão. Sidecar NÃO pode "diagnosticar TEA" via 1 keyword.

**Mitigação**:
- Mínimo de N sinais (3+ categorias diferentes?) pra alertar
- Confiança calibrada por Eduardo empíricamente
- Card sempre marcado como *"sinais detectados — médico interpreta"*

### 3. Cuidador vs paciente (semântica relacional)

Pra TEA infantil, falante = cuidador (mãe/pai), não paciente. Sidecar precisa capturar *"meu filho"* / *"minha filha"* / *"meu sobrinho"* — sinais REFERIDOS a terceiro.

Diferente de paciente adulto TEA falando de si.

**Janela contextual obrigatória**: pronome ambíguo (*"ele"*, *"ela"*) exige 3-5 frases anteriores pra resolver. Extrator recebe contexto, não frase isolada.

### 4. Volume pré-PMF baixo

Hoje provavelmente 0-2 casos neuro/TEA reais. Construir sidecar antes do volume real = risco de calibrar errado.

**Sequência empírica obrigatória**:
1. Eduardo traz 1-2 pacientes neuro REAIS
2. Análise manual dos primeiros 5-10 reports — quais sinais aparecem espontâneos?
3. Cristaliza lista empírica COM Eduardo (não chuta com base em literatura)
4. Codar sidecar depois

### 5. Conexão com V1.9.456 (já implementado 26/05)

O bloco "Histórico observacional" no modal do report já mostra padrão de queixas + sintomas novos. Pra Eduardo neuro, **isso já cobre 50% do que ele precisa SEM sidecar TEA**.

Quando o sidecar entrar, complementa: V1.9.456 mostra padrões + sidecar TEA mostra sinais comportamentais específicos categorizados.

### Fase C — Sidecar TEA quando Eduardo trouxer pacientes REAIS (V1.9.X+N, ~2h código)

Arquitetura técnica (modelo idêntico V1.9.307 renal + camada semântica):

```
1. Edge nova `tea-signal-extractor` (Deno, paralela)
2. Roda pós-AEC no Pipeline (NÃO bloqueia)
3. ALÉM disso: roda em chat_messages_legacy (chat livre) batch ou
   on-message — captura sinais espontâneos fora de AEC
4. Input: clinical_reports.content + chat_messages_legacy janela
5. GPT extraction (gpt-4o-mini, temp 0.1) com prompt restrito:
   "Extraia SOMENTE sujeito + comportamento + categoria.
    Diferencie sujeito DIRETO ('eu...') de REFERIDO ('meu filho...').
    NÃO infira diagnóstico. NÃO interpole.
    Se pronome ambíguo, resolva usando janela contextual N frases anteriores."
6. Output JSON estruturado:
   {
     "sujeito": "filho do paciente" | "próprio paciente" | "outro familiar",
     "comportamento": "morde tudo",
     "categoria": "agressividade_auto_lesao",
     "fala_literal": "meu filho morde tudo",
     "data": "2026-05-26T...",
     "etapa_aec": "lista_indiciaria",
     "confianca": "alta" | "media" | "baixa"
   }
7. Persistência: jsonb em clinical_reports.content.tea_signals OU tabela tea_signals
8. Card "Sinais TEA detectados" no dashboard Eduardo (allowlist neuro)
9. Card mostra: categoria + fala literal + data + sujeito identificado + confiança
10. Médico aprova/rejeita cada sinal (audit trail)
```

**Anti-regressão**:
- AEC FSM intocada
- CLINICAL_PROMPT intocado
- Verbatim First intocado
- Sidecar é paralelo (não bloqueia)
- Falha do sidecar não quebra Pipeline principal
- Audience Contract V1.9.330-A respeitado (médico interpreta, sidecar não diagnostica)
