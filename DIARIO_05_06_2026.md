## BLOCO C — 2 bugs que só o e2e pega (testes unitários não pegariam)

**A — Índices de idempotência PARCIAIS quebravam ON CONFLICT (V1.9.590).** Os índices do V1.9.587 eram `WHERE source_external_id IS NOT NULL`. O `supabase-js .upsert({onConflict})` gera `ON CONFLICT (col)` sem o predicado WHERE → Postgres não casa com índice parcial → "no unique constraint matching ON CONFLICT" → **todo insert de filho falhava**. Fix: índice unique COMPLETO (em coluna nullable, NULLs são distintos → nativos não colidem mesmo assim). Aplicado via PAT + migration no repo + reprovado e2e.

**B — Falso alarme honesto (corrigido em tempo real).** Primeiro diagnostiquei "duas CHECK constraints de gender contraditórias app-wide". **Errado** — foi artefato de query `WHERE conname='users_gender_check'` SEM filtro por tabela, que varreu a constraint homônima de uma tabela de **BACKUP** (`backup.backup_20251110_...`). A `public.users` viva tem **uma só** constraint: `gender IN (M,F,Outro)` (validada, igual ao que NewPatientForm grava). Pedro mandou *"investigar antes de tocar constraint"* — e a investigação mostrou que **não havia nada pra dropar**. O único problema real era o **meu mapper** emitindo `male/female/other` (formato do backup, não do app). Fix V1.9.591: `normalizeGender` → `M/F/Outro`.

## BLOCO D — Lições
1. **O piloto e2e justificou-se sozinho:** os 2 achados (índice parcial, formato de gender) são invisíveis a type-check/unit test — só aparecem com a Edge gravando de verdade. Mesma lição dos achados de 04/06 (FK auth.users, .limit(20)).
2. **Investigar antes de tocar (instinto do Pedro) evitou um DROP errado:** eu ia tratar uma constraint de backup como bug de produção. Filtrar `conrelid` por tabela desfez o falso alarme. Query de catálogo SEM filtro de tabela = pega homônimos de outros schemas (backup/legacy).
3. **Retorno de `.upsert` conta linhas *tentadas*, não *inseridas*** (com ignoreDuplicates) — idempotência só se prova **contando linhas persistidas via PAT**, não pelo retorno da função.

## BLOCO G — Paciente não enviava exame: RLS bloqueava INSERT (V1.9.594, bug Flávia)
Ricardo reportou que a Flávia (paciente real) não conseguia enviar exame. Print: *"Erro ao salvar metadados: new row violates row-level security policy for table patient_documents"*. **Não era tamanho nem HEIC** (hipóteses iniciais descartadas pelos dados) — era **descasamento frontend↔RLS**: `handleUpload` do `PatientMyExams` não setava `uploaded_by_role`, mas a policy "Patient INSERT own documents" tem `WITH CHECK (auth.uid()=patient_id AND uploaded_by_role='patient')`. Campo NULL → política barra → **100% dos pacientes** travados (os 6 docs existentes foram subidos pelo Ricardo via lado profissional, `uploaded_by_role='professional'` — workaround). Fix: insert seta `uploaded_by` + `uploaded_by_role='patient'`. **Provado nos 2 sentidos via RLS real** (`SET ROLE authenticated` + jwt sub Flávia, transação ROLLBACK): sem o campo reproduz o erro EXATO; com o campo passa. Fluxo inteiro checado sem regressão: upload ✅ + share (mesma exigência de campo) ✅ + médico vê (Flávia tem appointment com Ricardo) ✅. **Classe de bug: frontend que insere/atualiza precisa satisfazer o `WITH CHECK` da RLS** — campo que a política exige não pode ficar implícito.
