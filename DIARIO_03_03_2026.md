### 1. Análise da WiseCare API
- **v2.6.8**, REST API em `api.wisecare.com.br/api/v1`
- Hierarquia: Organization → Room → Session
- 23 endpoints: 9 Room + 10 Session + 4 Recording
- SDK npm: `wise-api` (import WiseApi from 'wise-api')
- Features: recordings, blockchain audit trails, participant management
- Auth: Basic Auth (login + password)
- 15 módulos no SDK: session, room, org, recording, transcription, etc.


## 👥 QUEM USA O WISECARE — POR TIPO DE USUÁRIO

### Onde o VideoCall é renderizado:

| Página | Usuário | WiseCare ativo? |
|:---|:---|:---|
| `AdminChat.tsx` | **Admin** | ✅ SIM — mesmo componente VideoCall |
| `PatientDoctorChat.tsx` | **Profissional / Paciente** | ✅ SIM — teleconsulta principal |
| `EduardoFaveretDashboard.tsx` | **Dr. Eduardo** | ✅ SIM — dashboard específico |

### Comportamento por tipo:

| Tipo de Usuário | O que acontece |
|:---|:---|
| **Profissional** | Inicia chamada → WiseCare cria sessão → iframe renderiza. Pode solicitar gravação clínica (5 min). |
| **Paciente** | Recebe notificação de chamada → aceita consentimento LGPD → entra na sessão WiseCare. Pode aceitar/recusar gravação. |
| **Admin** | Mesmo fluxo do profissional via AdminChat. Acesso completo incluindo video call. |
| **Todos** | Se WiseCare falhar → auto-fallback para WebRTC P2P sem intervenção do usuário. Badge indica qual provider está ativo. |

### Badge visual na tela:
- 🟢 Verde: "WiseCare" — telemedicina segura via infraestrutura profissional
- 🟡 Âmbar: "P2P Direto" — conexão direta (fallback, menos confiável em redes corporativas)
- ⚪ Cinza: "Conectando..." — tentando estabelecer conexão
